### Famille A : Cross-org leaks / IDOR (authz manquante)

| # | Route | Méthode | Problème | Fichier:ligne |
|---|-------|---------|----------|---------------|
| A1 | `/api/v1/approvals/[id]` | PATCH | **Aucune vérif** que user est délégué pour l'org ou admin → tout user approuve n'importe quelle demande | `src/app/api/v1/approvals/[id]/route.ts:5-36` |
| A2 | `/api/v1/approvals` | GET/POST | Pas de vérif user ∈ org. Query `?orgId=X` arbitraire retourne demandes de X | `src/app/api/v1/approvals/route.ts:9-53` |

| A4 | `/api/v1/asset-warranties` | GET | `?assetId=X` sans valider `asset.organizationId ∈ userOrgs` | `src/app/api/v1/asset-warranties/route.ts:9-24` |
| A5 | `/api/v1/asset-subscriptions` | GET | Idem A4 | même pattern |
| A6 | `/api/v1/asset-support-contracts` | GET | Idem A4 | même pattern |

| A9 | `/api/v1/software/licenses/[id]` | DELETE | Idem A7 | `src/app/api/v1/software/licenses/[id]/route.ts:29-34` |
| A10 | `/api/v1/policies/gpo-instances/[id]` | PATCH/DELETE | Pas de check orgId | `src/app/api/v1/policies/gpo-instances/[id]/route.ts:26-68` |
| A11 | `/api/v1/policies/documents/[id]` | PATCH | Pas de check orgId | `src/app/api/v1/policies/documents/[id]/route.ts:26-52` |

**Impact** : n'importe quel utilisateur authentifié (même CLIENT_USER/TECHNICIAN) peut lire, modifier ou supprimer des données d'autres organisations. Violation RGPD/Loi 25.

**Fix générique** : middleware `assertUserOrgAccess(me, orgId)` + `assertSameOrg(me, resource.organizationId)` systématique sur chaque route `[id]` et chaque GET filtré par ressource enfant.

## P1 — MAJEURS

### Pagination hardcodée (risque scale)

- `GET /api/v1/particularities` : `take: 200` sans skip/cursor (`route.ts:21-47`)
- `GET /api/v1/changes` : `take: 300` sans pagination (`route.ts:40`)
- `GET /api/v1/policies/{gpo,gpo-instances,documents}` : `take: 200` (`route.ts:31`)
- `GET /api/v1/approvals/inbox` : `take: 50` silencieusement tronque (`route.ts:35`)
- Engagements actifs (WarrantiesList/SubscriptionsList/SupportList) : tout le tableau chargé côté client sans pagination

### IA / Loi 25

- **M1-IA** : `runContentAssist()` appelé sans passer `aiConsent` ni vérifier `getAiConsent(orgId)` dans particularities (`[id]/ai/route.ts:48`, `bulk-apply/route.ts:50`)
- **M2-IA** : `analyzeGpoSource()` n'appelle jamais `getAiConsent(orgId)` avant Ollama (`src/lib/ai/gpo-analyze.ts:27-86`)
- **M3-IA** : `DEFAULT_CONSENT.aiEnabled = true` — orgs sans opt-in explicite sont analysées (`src/lib/ai/consent.ts:25-30`)
- **M4-IA** : `detectChangesForOrg()` aucun rate-limit/quota par org → cron agressif peut exploser coûts Ollama (`change-detect.ts:215-282`)
- **M5-IA** : `gpo-analyze.ts` ne distingue pas timeout Ollama de failure → payload reste `PENDING` sans retry possible

### Schéma / indexes manquants

- `GpoInstance` : index composite `(organizationId, templateId)` manquant pour requêtes de drift
- `ContactSoftwareAccess` : pas d'index sur `revokedAt` → full scan sur filtre `revokedAt: null`
- `ApprovalDelegate.scope` : `String @default("all")` au lieu d'enum → typos silencieux
- `ApprovalRequest.organizationId` : `String?` nullable → demandes orphelines contournent vérif org
- `ChangeAiSignal.status` : `String` au lieu d'enum
- `AssetSubscription.currency` : `String @default("CAD")` sans enum
- Pas d'unique constraint `AssetSubscription (assetId XOR softwareInstanceId)` → subs orphelines possibles

### Inter-modules / reverse lookups

- `ContentRelation` (schema:3042) **défini mais jamais utilisé** par Particularités/Software APIs
- Aucune route `GET /api/v1/policies/gpo-instances/[id]/approval` malgré `lastApprovalId` en schema
- `ScriptPublication.approvalId` stocké mais aucune route ne crée ScriptPublication après approval
- `/api/v1/tickets/[id]/changes?take=20` hardcodé → tickets très linkés perdent les anciens changements
- `AssetSubscription.autoRenew` : flag stocké mais **aucun service de renouvellement automatique** (gadget)

### Workflow / audit trail

- `ApprovalRequest` : pas de `ApprovalHistory` / audit trail → impossible de tracer qui a approuvé/rejeté/annulé
- `ApprovalRequest` : pas de `expiresAt` → demandes PENDING indéfiniment
- `ApprovalStatusP.CANCELLED` dans l'enum mais **jamais créé** par aucune route
- `PolicyDocument` : pas de versioning client (`templateSchemaVersion` interne uniquement)
- `SoftwareInstaller` : pas de `version`/`updatedAt` → remplacer un .msi perd la traçabilité
- Changes : `ChangeAiSignal` dismissed s'accumulent sans TTL/archivage (commentaire ligne 231 "ne pas re-classifier")

### Portail

- Route `/api/portal/renewals` vs `/api/v1/portal/renewal` : incohérence de nommage (page fetch `/api/portal/renewals?range=180d`)
- 7 × `dangerouslySetInnerHTML` dans pages portail sans sanitization visible serveur (policies, particularities, changes, software, tickets)
- Pas d'endpoint `/api/portal/search` dédié → recherche portail = impossible actuellement

## P2 — MINEURS

- Timeline changements : col DATE fixe 110px déborde mobile (`org-changes-tab.tsx:164-189`)
- `org-policies-tab.tsx` : **aucun** breakpoint Tailwind (sm/md/lg)

- Dossier PDF filename : `org.slug` directement interpolé, pas d'escape

- Dossier PDF : aucun logging/audit de téléchargement
- Maturité `computeMaturity()` : 14 requêtes `.count()/.findMany()` en Promise.all sans hints d'index
- Maturité : check `no_drifted_instances` n'inclut **pas** les changements (module nouveau) ni asset renewals
- Maturité portail : non exposée côté client (page/widget manquant)
- `decisionNote` approval : pas de limite longueur
- `CANCELLED` status : defined in enum, jamais créé → dead code

## Plan de remédiation proposé (Sprint 0 — bloquant prod)

1. **Semaine 1 — Fix IDOR/authz (Famille A)** : créer helper `assertSameOrg()`, appliquer sur les 12 routes identifiées. Tests intégration cross-org obligatoires.
2. **Semaine 1 — Fix visibility (Famille B)** : 
   - Wrap `/api/v1/search` avec filtre `visibility NOT IN ('INTERNAL')` sauf role SUPER_ADMIN/MSP_ADMIN
   - Ajouter `AND organization_id = ${orgId}` sur query articles KB
   - Filtrer INTERNAL dans `builder.ts` dossier PDF
   - Forcer check serveur sur toutes les routes `/api/portal/*`
3. **Semaine 2 — Secrets (Famille C)** :
   - Chiffrement at-rest `licenseKey` (AES-GCM + KMS ou lib sodium sealed boxes)
   - Nouveau secret obligatoire `NEXUS_REPORT_TOKEN_SECRET` (erreur boot si absent)
   - Rate limit 5 tentatives/PIN sur download links
4. **Semaine 2 — Pagination** : curseur opaque standard sur toutes les listes
5. **Semaine 3+ — Schema hardening** : enums manquants, indexes composites, ContentRelation câblé
