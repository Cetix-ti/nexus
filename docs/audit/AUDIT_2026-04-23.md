## 🟠 MAJEURS (à traiter)

### M1 — React hooks appelés conditionnellement (5 occurrences)

Bug React réel : violation de la règle des hooks. Peut causer crash ou état corrompu.

| Fichier | Ligne | Hook | Description |
|---|---|---|---|
| `src/app/(app)/organisations/[slug]/monthly-report-print/page.tsx` | 103, 108, 110 | `useState`, `useEffect` | appelé après `return` early |
| `src/components/ui/input.tsx` | 15 | `React.useId` | conditionnel |
| `src/components/ui/textarea.tsx` | 13 | `React.useId` | conditionnel |

**Fix** : déplacer tous les hooks en tête du composant, jamais après un `if`/`return`.

### M2 — React `set-state-in-effect` (78 occurrences)

Nouveau warning React 19 : setState dans useEffect sans dépendances claires peut causer des re-renders infinis ou des race conditions. La majorité sont des patterns `fetch → setState` légitimes, mais certains sont des vrais bugs.

Top fichiers à réviser en priorité :
- `src/app/(app)/organisations/[slug]/page.tsx` (5 occurrences : sites, contacts, tickets, contracts, dbOrg)
- `src/app/(app)/projects/[id]/page.tsx` (3 occurrences)
- `src/app/(app)/changes/[id]/page.tsx:48` (patch setter sans cleanup)
- `src/app/(app)/particularities/[id]/page.tsx:68` (idem)
- Tous les nouveaux détails pages (policies/*/[id], particularities/[id]) → même pattern générique à factoriser

**Fix** : extraire chaque `fetch+setState` dans un custom hook `useResource(url)` qui gère cancel + error.

### M3 — 254/366 routes API sans try/catch

69% des endpoints peuvent renvoyer 500 brut au lieu d'une erreur propre. Exemples où ça pique le plus :
- Routes qui font appel à Prisma (presque toutes) → `prisma.error` non capturée
- Routes qui parsent `req.json()` → crash sur JSON malformé
- Routes qui appellent des intégrations (freshservice, atera, ollama) → timeout = 500

**Fix recommandé** : wrapper `withApiError()` qui transforme toute exception en JSON `{error, code}`. Middleware simple à ajouter sur toutes les routes.

### m6 — `dangerouslySetInnerHTML` (20 occurrences)

Le HTML affiché vient de 2 sources :
1. **TipTap** (richtext editor) — sanitize par défaut côté client, safe
2. **Email entrants** (email-to-ticket) — passe par `sanitizeHtml` de `sanitize-html` (voir `src/lib/email-to-ticket/html.ts`)

Le HTML **stocké en DB depuis une source tierce non-TipTap** pourrait contenir du XSS. Risque faible car input contrôlé, mais recommandation : centraliser tout rendu HTML via un wrapper `<SafeHtml>` qui applique sanitize-html.

**Rien n'est cassé aujourd'hui**, mais si un jour on importe du HTML externe (webhook, API tierce) sans passer par TipTap, on aura une brèche XSS.

### i1 — IDOR pattern agent-facing

20 routes `v1/[id]/route.ts` utilisent `findUnique({id})` sans vérifier que le `me.id` (agent connecté) a le droit pour CETTE ressource.

**C'est le design MSP actuel** : tous les agents voient tous les clients. Si un jour on restreint (ex : agent Y ne voit que clients X et Z), ces 20 endpoints deviennent vulnérables.

### i2 — `$executeRawUnsafe` dans monitoring

`src/lib/monitoring/service.ts:241` utilise `executeRawUnsafe` avec `$1` / `$2` params → **SÛR**, paramétrage correct. Le "Unsafe" du nom est trompeur ici — l'API Prisma la nomme ainsi parce qu'elle ne valide pas le SQL côté Prisma, pas parce qu'elle est vulnérable.
